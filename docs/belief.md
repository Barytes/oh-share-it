# 产品定义与核心信念

## 1. 项目背景

在一个团队里，每个人都拥有自己的独特 context。

这里的 context 不只是文档、聊天记录或任务信息，而是一个人长期工作中沉淀下来的综合理解：他知道某个决策为什么被做出，知道某个客户真正关心什么，知道某段代码背后的历史包袱，知道某个流程为什么不能轻易改变，也知道哪些判断曾经被验证、哪些尝试曾经失败。

这些 context 往往来自个人的经历、观察、沟通、试错和判断。它们非常珍贵，却通常只存在于个人的大脑、私人笔记、零散对话、会议记忆和局部文档里。

当团队规模变大、项目变复杂、成员流动增加时，这些个人 context 会逐渐变成一个个孤岛：

- 新人很难快速获得老成员的经验，只能反复问人或重新踩坑。
- 跨团队协作时，大家掌握的信息不对称，导致沟通成本和误判增加。
- 关键成员离开后，他积累的判断、经验和历史理解很难完整留下。
- 团队虽然做了很多事，但真正能在合适场景被找到、理解和调用的 context 很少。
- AI 工具越来越强，但如果拿不到团队真实、细腻、及时的 context，也只能给出泛化答案。

因此，团队真正缺少的不是又一个知识库，而是一套能让个人 context 在保留边界的前提下被捕获、组织、路由和调用的系统。

## 2. 核心问题

团队知识管理长期存在一个根本矛盾：

一方面，团队中的重要知识大多分布在个人身上；另一方面，传统工具更擅长管理显性的、已经整理好的、静态的文档。

这导致大量有价值的 context 无法自然进入团队资产。

### 2.1 个人 context 难以被看见

很多重要信息并不会主动变成文档。它们藏在：

- 对某个问题的判断依据里。
- 对某个客户、用户、系统或市场的直觉里。
- 曾经讨论过但没有记录清楚的决策过程里。
- 一次失败实验背后的细节里。
- 某个人“知道不能这么做”的经验里。

这些知识不是没有价值，而是没有合适的容器。

### 2.2 分享 context 的成本太高

让一个人把自己的经验完整写成文档，通常是不现实的。

人们愿意回答具体问题，但很少有时间主动整理自己的全部知识；人们愿意在工作流里留下痕迹，但很难额外维护一个高质量知识库。

如果共享知识依赖大量手动整理，它就很难长期持续。

### 2.3 团队 context 难以持续积累

许多团队会在项目结束后写复盘，在决策后写纪要，在交接时写文档。但这些沉淀常常是一次性的、碎片化的、难以被再次调用的。

团队真正需要的是一种持续积累机制：每一次沟通、判断、产出、复盘，都能在合适的结构里变成未来可用的 context。

### 2.4 知识无法随着组织产生复利

优秀团队的优势不应该只来自当前在场的人，而应该来自过去所有人留下来的经验。

如果一个团队每次人员变化、项目重启、方向调整时都要重新理解一遍历史，那么团队就在反复支付认知成本。相反，如果过去的经验可以在正确权限和场景下被路由到当前问题中，团队就能少重复探索，多站在已有理解上继续前进。

## 3. 核心洞察

这个项目基于一个核心判断：

**未来的团队协作，不只是共享文件和任务，而是共享 context。**

文档回答的是“发生了什么”；context 回答的是“为什么是这样”“当时怎么想的”“现在应该如何判断”。

任务系统管理的是待办事项；context 系统管理的是理解、经验和判断。

知识库沉淀的是被整理后的结论；context 网络则保留知识产生的路径、来源、关系和适用边界。

当团队能够按场景路由 context 时，每个人不再只是访问一堆资料，而是能在需要时获得相关人的经验、判断、背景和不同视角。

## 4. 项目愿景

我们希望构建一个团队 context 路由系统，让每个人的经验在保留归属和边界的前提下被看见，让团队的知识能够自然沉淀，让过去的判断能够服务未来的决策。

这个系统应该帮助团队做到：

- 把个人独有的知识、经验、判断和历史理解转化为可共享的 context。
- 在不增加过多整理负担的情况下，让 context 从日常工作中自然沉淀。
- 让团队成员能够在需要时路由到相关的人、事、决策、经验和背景。
- 让离开的人的经验仍然对后来的成员有用。
- 让团队在每一次项目、协作和复盘后都变得更聪明。

最终，它应该成为团队的 context routing layer：不是替代个人思考，也不是把所有人的 context 捏成一个统一大脑，而是让每个人在合适边界下调用他人的理解，并保留自己的独立判断。

## 5. 核心价值

### 5.1 降低团队认知成本

团队成员不需要反复追问同样的背景，也不需要在零散聊天和文档中艰难拼图。

当一个人进入新项目、接手新模块、理解新客户或参与新决策时，他可以快速获得相关 context：历史发生过什么，关键约束是什么，谁最了解这个问题，哪些尝试已经做过，过去的判断依据是什么。

### 5.2 提升决策质量

很多错误决策并不是因为团队不聪明，而是因为 context 不完整。

当系统能够呈现相关历史、经验、反例、利益相关方、约束条件和曾经的判断过程时，团队就更容易做出成熟决策，减少重复试错。

### 5.3 保留组织记忆

团队最大的损失之一，是人的离开带走了大量隐性知识。

这个产品要帮助团队把个人经验转化为组织记忆，让经验不只依附于某一个人，而是能够成为团队长期可继承的资产。

### 5.4 加速新人融入

新人最难获得的不是流程说明，而是那些没人系统讲过的背景知识。

通过 context 共享，新人可以更快理解团队如何工作、为什么这么工作、哪些事情有历史包袱、哪些人拥有关键经验，从而更快做出有效贡献。

### 5.5 让 AI 真正理解团队

AI 的能力很大程度取决于它能获得什么 context。

如果团队的知识仍然分散在个人脑中和零散工具里，AI 只能基于通用知识回答问题。这个系统可以为 AI 提供团队专属的、高质量的、结构化的 context，让 AI 成为真正理解团队历史、业务、代码、客户和决策的协作者。

### 5.6 形成知识复利

每一次项目推进、问题解决、客户沟通、技术决策和复盘，都应该让团队未来更容易做类似事情。

这个产品的长期价值在于让团队知识不是线性消耗，而是在保留个人和局部边界的同时持续积累、关联、路由和增值。

## 6. 产品应该解决什么

这个产品不是简单的文档工具，也不是传统意义上的企业知识库。它需要围绕 context 的生命周期来设计。

### 6.1 捕获 context

从日常工作流中捕获有价值的信息，包括讨论、决策、项目过程、复盘、个人笔记、问答、文档和外部资料。

捕获的重点不是保存一切，而是识别哪些内容对未来理解和判断有价值。

### 6.2 组织 context

把零散信息组织成可理解、可检索、可关联、可路由的 context 网络。

它应该能连接人、项目、客户、代码、决策、问题、经验、文档和时间线，让用户看到知识之间的关系，而不是只看到孤立条目。

### 6.3 分享 context

让团队成员能够在合适的权限和场景下共享自己的 context。

分享不应该只是一篇文档，也可以是一个问题的回答、一段经验的沉淀、一次决策的解释、一个项目背景包，或某个主题下的个人判断集合。

### 6.4 路由和调用 context

用户应该能够在真正需要的时候，被路由到最相关、最可信、最合适边界的 context。

例如：

- 接手一个项目时，快速生成项目背景。
- 做技术决策时，查看相关历史和约束。
- 面对客户问题时，找到过去类似案例。
- 进行复盘时，回顾完整过程和关键判断。
- 向 AI 提问时，自动带上团队相关 context。

### 6.5 维护 context

context 会过期、变化、互相冲突。系统需要帮助团队管理这些变化。

它应该能提示哪些信息可能已经过时，哪些结论存在争议，哪些经验只适用于特定场景，哪些内容需要被确认或更新。

## 7. 产品原则

### 7.1 从人的 context 出发，而不是从文件出发

团队知识首先存在于人身上。产品应该尊重这一点，把“人知道什么”“人经历过什么”“人如何判断”作为重要入口。

### 7.2 降低表达成本

如果沉淀知识太麻烦，系统就会失败。

产品需要允许用户用最自然的方式贡献 context：提问、回答、摘录、语音、聊天、批注、复盘、同步已有工具，而不是要求每个人都成为知识库管理员。

### 7.3 保留背景，而不只保存结论

结论很重要，但只有结论往往不够。

好的 context 应该包含来源、时间、相关人、决策过程、适用条件和不确定性。这样未来的人才能判断它是否仍然可信、是否适用于当前场景。

### 7.4 让知识在使用中变好

知识系统不能只靠事后整理来维护。每一次搜索、提问、引用、确认、纠错和调用，都应该反过来帮助系统理解哪些 context 在哪些场景下更重要、更可信、更有用。

### 7.5 权限和信任必须内建

个人 context 可能包含敏感信息、未成熟判断或只适合局部共享的内容。

产品必须尊重权限、边界和来源透明。用户需要知道自己的 context 如何被使用，也需要知道自己看到的信息来自哪里、可信度如何。

### 7.6 路由优先，谨慎合并

产品不应该把所有人的 context 强行合并成一个统一知识体。

个人、项目、团队和公共层都可以保留自己的 context 空间。系统真正要做的是在具体问题发生时，根据场景、权限、来源、关系和意图，把合适的 context 路由给合适的人或 agent。

但这不意味着永远不能合并。稳定事实、低争议共识、标准流程、项目当前状态和公共入口，可以在保留来源、贡献链和可撤销机制的前提下，局部 merge 成公共层。

复用不是产品的根本动作，而是 context 被正确路由或谨慎合并后的结果之一。

### 7.7 Agent-facing 优先于 chatbot

AI 是这个系统的重要使用者，但产品不应该先被定义成一个聊天机器人。

更底层的产品能力应该是 agent-facing context layer：让人和 agent 都能稳定地检索、读取、引用、贡献、写回、同步和解释 context。微信、飞书、桌面聊天框或 MCP client 都只是入口，不应该决定知识库的结构。

## 8. 目标用户

### 8.1 高知识密度团队

适合研发、产品、设计、咨询、投资、销售、客户成功、研究、运营等依赖经验和判断的团队。

这类团队的价值不仅来自流程执行，也来自对复杂问题的理解、判断和历史积累。

### 8.2 快速成长的组织

当团队从几个人扩大到几十人、几百人时，知识孤岛会迅速变成瓶颈。

这个产品可以帮助快速成长的组织减少信息断层，保持协作质量。

### 8.3 人员流动带来高损耗的团队

如果一个人的离开会让项目、客户、代码或业务理解出现明显断层，这类团队会强烈需要 context 沉淀和继承。

## 9. 典型使用场景

### 9.1 新人加入项目

新人进入一个项目后，可以获得项目 context 包，包括项目目标、历史决策、关键人、主要风险、曾经踩过的坑、当前约束和推荐阅读路径。

### 9.2 接手离职成员的工作

接手者不仅能看到交接文档，还能看到离职成员在相关项目、客户、代码和决策上的经验沉淀，减少知识断层。

### 9.3 做重要决策

团队在做产品、技术或商业决策前，可以快速调用相关历史 context：过去是否讨论过，为什么当时没有做，做过哪些实验，谁参与过判断，现在条件是否变化。

### 9.4 解决复杂问题

当团队遇到一个问题时，系统可以帮助找到类似问题、相关经验、了解这个领域的人，以及过去被验证过的解决路径。

### 9.5 与 AI 协作

团队成员向 AI 提问时，系统可以自动带入相关 context，让 AI 不只是泛泛回答，而是基于团队真实情况给出建议、总结、风险提示和行动方案。

## 10. 项目的核心信念

1. 每个人都是团队知识的一部分，而不只是一个执行角色。
2. 最有价值的知识往往不是已经写成文档的知识，而是藏在经验、判断和背景里的 context。
3. 团队协作的下一阶段，是从共享信息走向共享理解。
4. 好的知识系统不应该要求人额外做大量整理，而应该从真实工作中自然沉淀。
5. 离开的人不应该带走团队全部记忆；留下来的经验应该继续帮助后来的人。
6. AI 要真正帮助团队，必须先理解团队自己的 context。
7. 团队知识的最终价值，不是被存起来，也不是被强行合并，而是在未来被正确路由和调用。

## 11. 一句话定义

这是一个帮助团队捕获、组织、共享、路由和调用个人 context 的系统，让每个人的经验在保留边界的前提下，成为团队可继承、可追溯、可增值的知识资产。

## 12. 未来可展开的问题

- 个人 context 的最小可分享单元是什么？
- 系统如何判断哪些 context 对未来有价值？
- 如何在自动沉淀和用户控制之间取得平衡？
- 如何设计个人 context、团队 context、项目 context 和公共路由层之间的关系？
- 如何让 context 具备可信度、时效性和适用边界？
- 如何让 AI 成为 context 的调用者、整理者和协作者？
- 如何处理敏感信息、未成熟判断和权限边界？
- 产品的第一个高频场景应该是新人 onboarding、项目交接、决策检索，还是 AI 问答？
