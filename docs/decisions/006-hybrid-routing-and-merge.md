# D-006：默认 routing，允许局部、可追溯、可撤销的 merge

**状态**：已采纳

**日期**：2026-05-01

## 背景

前面的讨论把两种极端路线区分开了：

- 把所有人的 context 捏合成一个统一公共知识体。
- 每个人完全保留独立 context，只在查询时做路由。

前者体验简单，但容易抹平个人判断、制造伪共识、破坏来源和边界。后者更尊重真实知识形态，但如果完全不做 merge，公共入口会弱，新人理解成本会高，系统也可能变成很多孤岛的索引。

因此，`oh-share-it` 不应该在这两者之间二选一。

## 决策

`oh-share-it` 采用混合策略：

**默认保留独立 context，用 routing 解决调用问题；当内容足够稳定、低争议、可验证、可追溯时，再允许局部 merge 成公共入口、共识页或项目记忆。**

也就是说，routing 是默认机制，merge 是有条件的晋升机制。

## 为什么不能默认 merge

默认 merge 会把 context 当成可以去人格化、去来源化的知识块。但很多高价值 context 不是这样。

一个人的判断往往依赖他的经历、角色、项目位置、信息来源和风险偏好。如果系统过早把这些判断合并成“团队答案”，就会丢掉最关键的信息：这个判断是谁在什么条件下形成的。

默认 merge 还会伤害用户信任。用户可能愿意让自己的 context 被引用、被请求、被路由给合适的人，但不一定愿意让它自动成为公共知识，甚至被改写成代表团队的结论。

## 为什么不能只做 routing

只做 routing 也有明显问题。

如果所有 context 永远停留在个人空间，用户每次都需要重新理解谁知道什么、哪些信息可信、哪些内容已经稳定。这会让新人 onboarding、项目交接、常见问题查询和标准流程学习变得很重。

一些内容确实应该被 merge：

- 稳定事实。
- 团队共识。
- 标准流程。
- 术语定义。
- 项目当前状态。
- 多人反复确认过的背景。
- 已经不依赖个人视角的公共入口。

这些内容如果始终只靠 routing，会让系统显得没有公共记忆。

## Merge 的适用条件

内容进入公共 merge 层前，至少应满足其中几类条件：

- **稳定**：短期内不太可能频繁变化。
- **低争议**：没有明显冲突观点，或冲突已经被明确标注。
- **可验证**：能回溯到来源、证据或责任人。
- **可归属**：知道是谁贡献、确认或维护的。
- **可撤销**：未来发现不准确时，可以降级、修订或拆回多个 context。
- **有公共入口价值**：能帮助多人快速建立背景，而不是只服务某个局部问题。

## Routing 的适用场景

以下内容默认应保留独立 context，并通过 routing 使用：

- 个人判断。
- 未成熟想法。
- 局部经验。
- 角色相关视角。
- 项目内部细节。
- 敏感或半敏感信息。
- 仍在变化的策略、计划和判断。
- 存在明显 tension 的观点。

这些内容不应该为了“复用”被强行公共化。系统应做的是让它们在合适的权限和问题下被找到、引用和解释。

## 产品形态

可以把系统分成四类 context 空间：

1. **Personal Context**
   个人笔记、判断、经验、私有材料、未成熟想法。

2. **Project Context**
   某个项目内共享的背景、决策、实验、交接、问题历史。

3. **Routed Context**
   系统根据问题、权限、关系和来源，临时组合出来的上下文。

4. **Public / Consensus Context**
   被确认稳定、适合长期共享的团队共识、流程、术语、公共入口。

这四层不是固定等级，而是不同边界。一个 context 可以先存在于个人空间，被路由给项目成员，后来被整理进项目记忆，最终其中一部分晋升为公共共识。

## 对 agent 的含义

agent 不应该默认问一个统一知识库要答案，而应该先判断当前任务需要哪类 context：

- 需要公共事实时，优先查 public / consensus context。
- 需要项目背景时，查 project context。
- 需要某个人的经验和判断时，请求或引用 personal context。
- 遇到冲突时，呈现 tension，而不是强行合成。
- 需要行动建议时，明确说明引用了哪些 context、哪些仍不确定。

这意味着 agent-facing tool surface 需要支持的不只是 search，还包括 routing、source explanation、scope filtering、merge proposal 和 tension detection。

## 设计约束

- Merge 后的内容必须保留来源和贡献链。
- Merge 后的内容必须允许修订、降级或拆分。
- 系统不应把被频繁路由等同于应该 merge。
- 公共层可以保存摘要、索引和入口，不必复制所有个人 context 全文。
- 存在冲突时，优先保留 tension；只有在冲突被解决或范围被澄清后，才合并成公共结论。

## 一句话

`oh-share-it` 的核心策略不是 merge everything，也不是 route everything，而是：

**默认 routing，谨慎 merge；让 context 保持边界，同时让稳定知识拥有公共入口。**
