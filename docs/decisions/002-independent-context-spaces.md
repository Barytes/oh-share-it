# D-002：保留每个人独立的 context，不把所有 context 强行合并

**状态**：已采纳

**日期**：2026-05-01

## 背景

早期讨论中，“复用”和“公共知识资产”容易被理解成：把每个人的 context 提炼、合并、压缩成一个统一的团队知识体。

这会带来糟糕的产品倾向：

- 过早抹平个人判断差异。
- 把未成熟想法、局部经验和个人视角错误公共化。
- 让用户担心自己的 context 被系统拿去合并、改写或代表自己。
- 把有价值的 tension 误处理成共识。
- 为了追求复用而牺牲 context 的来源、边界和适用条件。

## 决策

`oh-share-it` 不追求把所有人的 context 捏成一个共享大脑。

每个人、项目、团队和公共层都可以保留自己的 context 空间。产品要解决的是在具体问题发生时，如何根据场景、权限、来源、关系和意图，把合适的 context 路由给合适的人或 agent。

## 产品含义

- “共享”不是默认合并，而是可控暴露、可追溯引用和按需调用。
- “复用”不是产品的根本动作，而是 context 被正确路由后的结果之一。
- 个人 context 可以被引用、摘要、推荐或参与回答，但不应自动失去归属和边界。
- 公共层应更像索引、路由和治理层，而不是吞并个人 context 的中央仓库。

## 设计约束

- context 必须保留 owner、source、scope、visibility、updated_at 等边界信息。
- 同一问题下可以存在多个人的不同 context，不必强行合成单一答案。
- 当多个 context 互相冲突时，系统应优先呈现 tension、来源和适用条件，而不是过早合并。
